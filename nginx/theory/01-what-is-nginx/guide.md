# 1. What is Nginx, and what is the difference between using it as a **web server** vs a **reverse proxy** vs a **load balancer**?

## 2. Interviewer's Point of View

The interviewer wants to assess whether you understand **Nginx's core capabilities** and can articulate **when and why** to deploy it in each role.  They are looking for:
- A clear mental model of the request flow.
- Awareness of the trade‑offs (performance, flexibility, TLS termination, health‑checking, etc.).
- Ability to map real‑world scenarios (static site, API gateway, traffic distribution) to the appropriate Nginx configuration.

## 3. Steps to Set Up a Minimal Example for Each Role

> *Thoughts to share:* “I’ll spin up three tiny Docker containers, each exposing a static HTTP service, and point Nginx at them in different ways.”

1. **Web Server** – Serve static files directly.
   ```bash
   docker run -d --name nginx-web -p 8081:80 -v $(pwd)/static:/usr/share/nginx/html nginx:latest
   ```
2. **Reverse Proxy** – Proxy to an upstream app (e.g., a simple `httpd` container).
   ```bash
   docker run -d --name upstream-app -p 8082:80 httpd:latest
   docker run -d --name nginx-proxy -p 8083:80 -v $(pwd)/proxy.conf:/etc/nginx/conf.d/default.conf nginx:latest
   ```
3. **Load Balancer** – Distribute traffic across two identical upstream apps.
   ```bash
   docker run -d --name upstream-a -p 8084:80 httpd:latest
   docker run -d --name upstream-b -p 8085:80 httpd:latest
   docker run -d --name nginx-lb -p 8086:80 -v $(pwd)/lb.conf:/etc/nginx/conf.d/default.conf nginx:latest
   ```

*Note:* The actual configuration files (`proxy.conf`, `lb.conf`) are tiny snippets demonstrating the three roles; they are omitted here for brevity.

## 4. Conceptual Explanation (Not Debugging – Theory Focus)

| Role | Primary Function | Typical Use‑Case | Key Directives |
|------|------------------|-----------------|----------------|
| **Web Server** | Serves static content directly from the file system. | Hosting static websites, assets, SPA builds. | `root`, `index`, `try_files`.
| **Reverse Proxy** | Accepts client requests and forwards them to an upstream service, optionally terminating TLS. | API gateway, micro‑service front‑end, TLS termination. | `proxy_pass`, `proxy_set_header`, `ssl_certificate`.
| **Load Balancer** | Distributes incoming connections across a pool of upstream instances using a balancing algorithm. | Scaling services, high‑availability, blue‑green deployments. | `upstream` block, `proxy_pass` to the upstream, `load_module` (e.g., `least_conn`, `ip_hash`).

**Key Differences**
- **Network Hop**: Web server → single hop; reverse proxy → one hop to upstream; load balancer → multiple possible upstreams.
- **TLS**: Only reverse proxy/load balancer typically terminate TLS; web server can serve TLS but then it is just a web server.
- **Health‑checking**: Load balancer can perform health checks (`max_fails`, `fail_timeout`). Reverse proxy usually trusts the upstream.
- **Performance Overhead**: Adding a proxy or balancer introduces an extra TCP round‑trip, but gains flexibility.

## 5. Interview Summary Pitch

> *“Nginx is a versatile HTTP engine that can act as a simple static web server, a reverse‑proxy that terminates TLS and forwards requests to backend services, or a load balancer that distributes traffic across multiple upstreams using algorithms like round‑robin or least‑connections. The choice depends on whether you need to serve static files directly, centralise routing/TLS for micro‑services, or scale those services horizontally. In practice I would start with a reverse‑proxy configuration for API services and then add an upstream pool with a load‑balancing strategy as traffic grows.”*
